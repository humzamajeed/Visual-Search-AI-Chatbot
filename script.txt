🎬 SEGMENT 3 — Docker & Containerization (~2.5 minutes)
> "Now the most important foundation of our pipeline — Docker.
>
> So what is Docker? Simply put — Docker packages your application into a container. A container has everything your app needs to run — the code, the libraries, the settings. So it runs the same way on every machine. No more 'it works on my laptop but not on the server' problem.
>
> Let me show you our Dockerfile."
```bash
cat backend/Dockerfile
```
> "This is the Dockerfile for our backend. It installs Python, then installs PyTorch and FAISS — the libraries our AI search needs. Our frontend has a separate Dockerfile that uses a multi-stage build — first it compiles the React app, then it serves it using Nginx. Multi-stage means the final image is smaller and cleaner.
>
> Now — we don't run these containers one by one. We use Docker Compose to run all three together."
```bash
docker compose up -d
```
> "This one command starts our entire application — MongoDB, backend, and frontend — all together, on the same network, with shared storage for the database."
```bash
docker compose ps
```
> "All three services are running. But let me prove this is actually working — not just containers sitting there doing nothing."
```bash
curl -s http://127.0.0.1:5000/signup -X POST -H "Content-Type: application/json" -d '{"username":"fdemo","password":"fdemo123"}'
echo ""
```
> "We just created a new user. Now let me log in."
```bash
curl -s http://127.0.0.1:5000/login -X POST -H "Content-Type: application/json" -d '{"username":"fdemo","password":"fdemo123"}'
echo ""
```
> "You can see a JWT token in the response. This is a real login — the request went from the terminal, to the Flask backend, to MongoDB, and came back with a token. Everything is connected and working.
>
> Now let me shut this down because in the next step, we will deploy this properly using Kubernetes."
```bash

```

🎬 SEGMENT 5 — Kubernetes Orchestration (~3.5 minutes)
> "Now let's talk about Kubernetes — this is the biggest part of what I worked on.
>
> So, Docker gives us containers. But in real production, you don't run just one container — you need many containers working together, you need them to restart automatically if they crash, you need to scale up when traffic increases. That's exactly what Kubernetes does. It's a system that manages all our containers for us.
>
> Let me prove something important first — that our deployment works completely from scratch, with zero manual steps."
```bash
minikube status
kubectl delete namespace visual-search
kubectl get namespaces
```
> "I just deleted everything. Our entire application is now completely gone from this cluster. Watch what happens next."
```bash
./start.sh
```
> "This single script — start dot sh — brings back our entire application by itself. While this rns, let me explain what it's doing. It creates a namespace, which is like a separate workspace for our app. Then it applies our secrets — our database passwords, securely stored, never written in plain text in our code. Then it sets up persistent storage for our database, so our data survives even if a container restarts. Then it deploys MongoDB, then our backend, then our frontend, then our monitoring tools.
>
> One more thing — this script is self-healing. If our database storage was left in a strange state from a previous run, the script automatically detects that and fixes it before continuing. We added that after running into this exact problem ourselves during testing."
[Wait for it to finish]
```bash
kubectl get pods -n visual-search
kubectl get svc -n visual-search
```
> "And there it is — five pods running. MongoDB, backend, frontend, Prometheus, and Grafana. Each one has its own Service, which is how they find and talk to each other inside the cluster."
```bash
kubectl get pv
kubectl get pvc -n visual-search
```
> "And here's our persistent storage — bound and ready. This means our database data is safe even if pods restart."
> "Now let me show you something really powerful — self-healing."
```bash
kubectl delete pod $(kubectl get pods -n visual-search -l app=backend -o jsonpath='{.items\\\[0].metadata.name}') -n visual-search
kubectl get pods -n visual-search -w
```
> "I just deleted our backend pod on purpose. Watch — Kubernetes notices it's gone and creates a brand-new one automatically, within seconds. In a real production system, this means if a server crashes at 3 AM, nobody needs to wake up and fix it manually."
[Ctrl+C once the new pod shows Running]
> "Next — scaling. If our app suddenly gets a lot of traffic, we can handle it instantly."
```bash
kubectl scale deployment backend -n visual-search --replicas=3
kubectl get pods -n visual-search
```
> "One command, and now we have three copies of our backend running at the same time, sharing the load.
>
> Last thing — updates. What happens when we want to deploy a new version of our app?"
```bash
kubectl set image deployment/backend backend=docker.io/humza1535/visual-search-backend:v2 -n visual-search
kubectl rollout status deployment/backend -n visual-search
```
> "This is called a rolling update. Kubernetes replaces our old pods with new ones, one at a time, so our app never goes down during the update. And if something goes wrong with the new version —"
```bash
kubectl rollout undo deployment/backend -n visual-kubectl rollout undo deployment/backend -n visual-search
```
> "— we can roll back instantly to the previous working version. This is a really important safety feature for real production systems."
```bash
kubectl scale deployment backend -n visual-search --replicas=1
```

> Here's the problem we wanted to solve — once an application is running in production, how do you actually know if it's healthy? How many requests is it getting? Are there errors? Without monitoring, you're completely blind. So we set up Prometheus and Grafana.

Prometheus automatically scrapes that endpoint every few seconds. And here in Grafana, you can see real numbers — the actual requests we just sent, broken down by status code. This gives us real visibility into how our application is behaving, live."

🎬 SEGMENT 7 — Infrastructure as Code with Terraform & AWS (~2 minutes)
> "Now let me show you our cloud deployment work. This was a bonus part of our project.
>
> So far everything we've shown runs locally on our own machine using Minikube. But real production systems run on the cloud — AWS, Azure, Google Cloud. The old way of setting up cloud servers is clicking around manually in a web console — slow, and easy to make mistakes or forget what you did. So instead, we used Terraform. Terraform lets us write our cloud infrastructure as code — meaning it's written down, version controlled, and repeatable.
>
> Let me show you our Terraform file."
```bash
cd terraform
cat main.tf
```
> "This file describes exactly what we want on AWS — a security group for network rules, and a virtual server, called an EC2 instance. Let's see what Terraform plans to do before actually doing it — this is an important safety practice, you always preview changes first."
```bash
terraform plan
```
> "It shows exactly two things it will create. Nothing is hidden, nothing is surprising. Now let's actually create it."
```bash
terraform apply
```
> "While this runs — it's creating a real virtual machine on Amazon's cloud right now, fully automatically, from this one command."
```bash
aws ec2 describe-instances --filters "Name=tag:Name,Values=visual-search-devops-project" --query 'Reservations\\\[].Instances\\\[].{ID:InstanceId,State:State.Name,IP:PublicIpAddress}' --output table
```
> "And here it is — a real, running server on AWS, with a real public IP address. This proves our infrastructure as code actually works on a real cloud provider, not just locally.
>
> Now, since this is just a demonstration and we don't want to leave it running and accumulating any cost, let's clean it up properly."
```bash
terraform destroy
```
> "And it's gone. This is responsible cloud resource management — never leave things running that you don't need. That's an important security and cost practice in real DevOps work."

🎬 SEGMENT 8 — Issues We Faced During Development (~2 minutes)
[All three, one after another]
> \\\*\\\*Humza:\\\*\\\* "I want to share some real problems we ran into. First — our backend code needed an AI model file and a search index file that were too large to store in Git, so they were never actually in our repository. When we tried to run the app fresh, it crashed immediately on startup. We fixed this by writing a stub version of that module — so the app boots up normally and all the other features work fine, even without the full AI model loaded."
> \\\*\\\*\\\[Teammate 2]:\\\*\\\* "I ran into a tricky Kubernetes bug. In our configuration file, we had a line that builds our database connection string using two other variables — a username and a password. But I had accidentally written that line \\\*before\\\* defining those two variables in the file. Kubernetes reads these top to bottom, so it couldn't substitute the values, and our backend kept crashing with login errors. I found this by reading MongoDB's own error logs, which said it couldn't find a user with a strange-looking name — that's when I realized the variables weren't being replaced properly. Moving those two lines above fixed it completely."
> \\\*\\\*\\\[Teammate 3]:\\\*\\\* "I learned that Grafana dashboards aren't saved permanently by default — when we deleted and recreated our Kubernetes setup during testing, our dashboard disappeared completely because we hadn't given Grafana its own persistent storage like we did for MongoDB. I had to rebuild it for this recording. We've noted that as something to fix properly going forward. I also ran into some tricky networking and free-tier server type issues on AWS, which took some research and trial and error to resolve."
